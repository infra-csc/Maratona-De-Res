Crie um aplicativo web completo chamado **Maratona de Resultados – Cenográfica**.

O objetivo do app é avaliar trimestralmente os colaboradores da função **Cenográfica** que trabalham nos eventos da empresa, calcular a média ponderada por evento, consolidar o resultado por trimestre, aplicar descontos por faltas e classificar os colaboradores em pelotões com bônus no cartão Caju.

Este app precisa ser funcional, bonito, moderno e pronto para evoluir. Não quero apenas um protótipo visual. Quero banco de dados, autenticação, regras, cálculos, telas, validações, seed inicial e dashboard.

# 2. Regra de negócio principal

O app avalia colaboradores da área **Cenográfica** por evento.

Cada evento possui colaboradores participantes.

Cada colaborador é avaliado em até **7 quesitos**, mas nem todos os eventos usam todos os quesitos.

Cada quesito possui um peso definido pelo RH. A soma padrão dos pesos é **20**.

Quando um evento não usa todos os quesitos, os pesos dos quesitos ativos no evento devem ser redistribuídos proporcionalmente para continuar fechando em **20**.

Cada avaliador informa uma nota de **0 a 5**.

A Diretoria pode calibrar a nota depois das avaliações. Quando existir nota calibrada, ela substitui a média original no cálculo final.

A consolidação é feita por **trimestre**.

Cada falta desconta pontos do resultado final do trimestre. O valor inicial é **50 pontos por falta**, mas deve ser editável na tela de regras.

# 4. Cálculos obrigatórios

Crie funções utilitárias no back-end para os cálculos. Essas funções precisam ser reutilizáveis e fáceis de testar.

## 4.1 Conversão da nota

A nota do avaliador é de 0 a 5.

Converter para escala de 0 a 100:

```txt
nota_percentual = (nota / 5) * 100
```

Exemplos:

```txt
5 = 100
4 = 80
3 = 60
2 = 40
1 = 20
0 = 0
```

---

## 4.2 Normalização dos pesos

A soma oficial dos pesos deve ser 20.

Se o evento usar todos os quesitos, usar os pesos normais.

Se o evento usar apenas alguns quesitos, recalcular os pesos proporcionalmente.

Fórmula:

```txt
peso_normalizado = (peso_original / soma_pesos_ativos_do_evento) * 20
```

Exemplo:

Pesos originais:

```txt
Qualidade da Entrega = 4
Cumprimento de Prazo = 4
Organização e Planejamento = 3
Trabalho em Equipe = 3
Proatividade = 2
Comunicação = 2
Postura e Segurança = 2
Total = 20
```

Se o evento usar só os 4 primeiros:

```txt
Soma ativa = 4 + 4 + 3 + 3 = 14
```

Os pesos devem ser redistribuídos para fechar 20.

---

## 4.3 Nota usada no cálculo

Para cada colaborador, evento e quesito:

1. Se existir nota calibrada pela Diretoria, usar a nota calibrada.
2. Se não existir nota calibrada, usar a média das notas enviadas pelos avaliadores.
3. Se não houver avaliação enviada, marcar como pendente.
4. Não permitir fechamento normal do evento com pendências.
5. Admin/Diretoria podem forçar fechamento, mas devem informar justificativa.

---

## 4.4 Resultado do evento

Para cada colaborador em cada evento:

```txt
resultado_evento = soma(nota_percentual_do_quesito * peso_normalizado_do_quesito) / 20
```

O resultado deve ficar entre 0 e 100.

---

## 4.5 Resultado trimestral

O resultado trimestral considera os eventos fechados dentro do trimestre.

Regra padrão:

```txt
media_bruta_trimestre = média simples dos resultados dos eventos do trimestre
```

Deixe o código preparado para futuramente permitir peso por evento:

```txt
media_bruta_trimestre = soma(resultado_evento * peso_evento) / soma(peso_evento)
```

Na primeira versão, todo evento tem peso 1.

---

## 4.6 Desconto por faltas

Cada falta desconta pontos do trimestre.

Valor inicial:

```txt
1 falta = -50 pontos
```

Fórmula:

```txt
desconto_faltas = quantidade_faltas * pontos_por_falta
resultado_final = media_bruta_trimestre - desconto_faltas
```

O resultado final não pode ser menor que 0 nem maior que 100.

```txt
resultado_final = min(100, max(0, resultado_final))
```

---

## 4.7 Pelotões e bônus

Crie classificação configurável.

Regras iniciais:

```txt
Pelotão Quênia:
resultado >= 90
bônus = R$ 3.200

Pelotão Azul:
resultado >= 80 e resultado < 90
bônus = R$ 2.400

Pelotão Verde:
resultado >= 70 e resultado < 80
bônus = R$ 1.600

Pelotão Branco:
resultado < 70
bônus = R$ 0
```

Essas regras devem ser editáveis na tela de Regras.

Importante: evitar lacunas. Toda nota de 0 a 100 precisa cair em algum pelotão.
