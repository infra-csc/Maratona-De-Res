Agora preciso ajustar o app com base no modelo real de avaliação que usamos hoje.

Importante: **a planilha enviada é apenas uma referência de regra de negócio, quesitos, pesos, notas, comentários e cálculo. Não quero que o app copie o visual da planilha.**

O app deve manter a lógica da planilha, mas transformar isso em uma experiência web moderna, visual, profissional e muito melhor que Excel.

Não recomece o projeto do zero. Continue a partir do app **Maratona de Resultados – Cenográfica** já criado.

---

# 1. O que a planilha representa

A planilha mostra como a avaliação é feita hoje.

Ela possui:

* Nome do evento
* Colaboradores avaliados
* Resultado por colaborador
* Quesitos
* Campo “O que é analisado”
* Peso
* Nota final
* Total
* Comentários das notas

Use isso como referência funcional, não como referência visual.

O app não precisa parecer uma planilha.

---

# 2. Quesitos fixos da avaliação

Os quesitos da avaliação são fixos para os eventos.

Criar estes 7 quesitos como base principal do sistema:

```txt
Perda de material/estrutura
Ferramentas & case
Qualidade da entrega
Qualidade técnica da montagem
Logística reversa
Prazo da entrega
Conduta e comportamento
```

Na planilha existe um item chamado **Qualidade da entrega1**. No app, esse item deve ser tratado como:

```txt
Qualidade técnica da montagem
```

Isso evita nome duplicado e deixa o processo mais claro.

---

# 3. Campo “O que é analisado”

O campo **“O que é analisado”** também deve ser fixo para cada quesito.

Criar as descrições abaixo:

```txt
Perda de material/estrutura:
Todo material enviado deve retornar à base sem perda de mercadorias, materiais ou avarias.

Ferramentas & case:
Todas as ferramentas e cases devem retornar à base corretamente.

Qualidade da entrega:
Avalia acabamento, materiais em bom estado e qualidade visual da entrega.

Qualidade técnica da montagem:
Avalia se a montagem foi executada corretamente, se houve problemas estruturais, necessidade de ajustes em arena ou falhas que impactaram a entrega.

Logística reversa:
Avalia se a carga foi feita adequadamente e conforme o alinhamento combinado.

Prazo da entrega:
Avalia se as entregas ocorreram dentro do cronograma, sem atrasos e sem custos adicionais de mão de obra.

Conduta e comportamento:
Avalia uso de uniforme, EPI, envio de comprovações e fotos, horários na arena, comportamento profissional e cuidado com ferramentas.
```

Essas descrições devem aparecer para orientar o avaliador no momento da avaliação.

# 4. Pesos fixos iniciais

Usar os pesos atuais da planilha:

```txt
Perda de material/estrutura = 3
Ferramentas & case = 3
Qualidade da entrega = 2
Qualidade técnica da montagem = 3
Logística reversa = 3
Prazo da entrega = 3
Conduta e comportamento = 3

Total = 20
```

Por padrão, todos os eventos devem usar os 7 quesitos.

Mesmo assim, manter a estrutura técnica preparada para desativar algum quesito em um evento específico no futuro.

Caso algum quesito seja desativado em um evento, o app deve redistribuir proporcionalmente os pesos dos quesitos ativos para continuar fechando em 20.

---

# 5. Regra de cálculo correta

A regra atual é:

```txt
Total do quesito = Nota usada x Peso
Resultado do evento = Soma dos totais dos quesitos
```

Como a soma dos pesos é 20 e a nota máxima é 5:

```txt
20 x 5 = 100 pontos
```

Portanto, o resultado do evento já fica naturalmente em uma escala de 0 a 100.

Não dividir novamente por 20 se o cálculo já estiver usando:

```txt
nota x peso
```

---

# 6. Nota usada no cálculo

Para cada colaborador, evento e quesito:

```txt
Se existir nota calibrada pela Diretoria:
usar nota calibrada.

Se não existir nota calibrada:
usar a nota original/média dos avaliadores.
```

Nomenclatura oficial do app:

```txt
Nota Original = média das notas dos avaliadores
Nota Calibrada = nota ajustada pela Diretoria
Nota Usada = nota que entra no cálculo
Total do Quesito = Nota Usada x Peso
Resultado do Evento = soma dos totais dos quesitos
Resultado Trimestral = média dos eventos - desconto por faltas
```

---

# 7. Exemplo obrigatório para validar o cálculo

Criar um teste ou validação no código com este exemplo:

```txt
Pesos = [3, 3, 2, 3, 3, 3, 3]
Notas = [4, 4, 4, 3, 2, 3, 5]
```

Cálculo esperado:

```txt
3 x 4 = 12
3 x 4 = 12
2 x 4 = 8
3 x 3 = 9
3 x 2 = 6
3 x 3 = 9
3 x 5 = 15

Resultado = 71
```

O app precisa retornar exatamente:

```txt
71
```

Se retornar diferente de 71, o cálculo está errado.

# 13. Experiência do avaliador

O avaliador não deve preencher uma planilha.

Criar uma experiência guiada:

1. Seleciona o evento.
2. Visualiza os colaboradores pendentes.
3. Seleciona um colaborador.
4. O app mostra os 7 quesitos fixos.
5. Para cada quesito, o avaliador informa nota de 0 a 5.
6. O avaliador preenche comentário.
7. Comentário obrigatório quando nota for menor que 3.
8. O app calcula automaticamente o total de cada quesito.
9. O app mostra o resultado parcial em tempo real.
10. O avaliador pode salvar rascunho.
11. O avaliador pode enviar avaliação final.

Após envio, bloquear edição para perfil Avaliador.

Admin pode reabrir avaliação.

# 16. Resultado trimestral

O resultado trimestral continua sendo:

```txt
Resultado trimestral = média dos resultados dos eventos do trimestre - desconto por faltas
```

Cada falta desconta o valor configurado na tela de Regras.

Valor inicial:

```txt
1 falta = -50 pontos
```

O resultado final nunca pode ficar abaixo de 0 nem acima de 100.

```txt
resultado_final = min(100, max(0, resultado_final))
```

# 17. Pelotões e bônus

Usar a classificação abaixo:

```txt
Pelotão Quênia:
resultado >= 90
bônus Caju = R$ 3.200

Pelotão Azul:
resultado >= 80 e resultado < 90
bônus Caju = R$ 2.400

Pelotão Verde:
resultado >= 70 e resultado < 80
bônus Caju = R$ 1.600

Pelotão Branco:
resultado < 70
sem bônus
```

Essas regras devem continuar editáveis por Admin/RH na tela de Regras.

Não permitir lacunas entre as faixas.

# 22. Critérios de aceite deste ajuste

Este ajuste estará correto se:

1. O app não copiar o visual da planilha.
2. A planilha for usada apenas como referência de regra.
3. Os 7 quesitos forem fixos.
4. O campo “O que é analisado” for fixo para cada quesito.
5. Os pesos iniciais forem iguais ao modelo atual.
6. O cálculo for `nota usada x peso`.
7. O resultado do evento for a soma dos totais dos quesitos.
8. O exemplo da planilha retornar exatamente 71.
9. A tela de avaliação for moderna, visual e profissional.
10. O avaliador conseguir preencher notas e comentários facilmente.
11. Comentário for obrigatório para nota menor que 3.
12. A Diretoria conseguir calibrar notas.
13. A nota calibrada substituir a nota original.
14. Dashboard, ranking e trimestre usarem esse cálculo.
15. O resultado trimestral descontar faltas.
16. Os pelotões e bônus continuarem funcionando.
17. Todos os dados estiverem conectados ao banco.
18. Nenhum botão principal ficar sem função.

# 23. Muito importante

Não transformar a tela em uma cópia de Excel.

A experiência deve ser melhor do que a planilha atual.

A planilha serve para mostrar o processo atual, mas o app precisa entregar uma visualização mais clara, mais bonita, mais rápida e mais confiável.

O objetivo é substituir o Excel por um sistema web profissional para a **Maratona de Resultados – Cenográfica**.
