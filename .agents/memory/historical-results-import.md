---
name: Historical results import (pre-calibrated events)
description: How events with a pre-final, already-calibrated score (no per-criterion evaluation) bypass the normal scoring pipeline
---

Some events (e.g. imported historical race results) arrive with a single final score per team/employee — there are no per-criterion evaluations, calibrations, or conformity penalties to compute.

**Why:** the existing pipeline (`computeEventTeamResult`) assumes raw evaluations + weighted criteria + conformity penalties exist. Historical data only has a final number, so forcing it through that pipeline would require fabricating fake evaluation rows.

**How to apply:** model this as an event-level flag (`isHistorical` + `importedScore` on the events table), not a separate data path. Every place that computes/reads event results (`recomputeCycleResults`, `GET /events/:id/result`, `GET /events` list, `loadEventDetail`) must special-case `isHistorical` events to use `importedScore` directly as `eventScore = calibratedEventScore = finalEventScore`, with `evaluationProgress = 1` and `hasEvaluations = true`, bypassing `computeEventTeamResult` entirely. Missing any one of these call sites causes historical events to silently show as 0%/unevaluated in only that view.

Import endpoint pattern (bulk CSV/TSV import with dry-run):
- Always require an explicit `dryRun: true` preview pass before allowing commit; the same validation/matching code path must run for both dry-run and commit so the preview is trustworthy.
- Group input rows by (event name + date) since one event has many employee rows.
- Employee matching: normalize (NFD strip + lowercase + trim) and require an exact match. A name with zero matches is NOT a hard error — auto-create the employee record (name only, defaults for the rest) at commit time and link it as a participant; only a name matching 2+ existing employees (genuinely ambiguous, can't guess which) is a hard error requiring manual resolution.
- Detect three actions per event group: create (new), update (existing event already `isHistorical`), conflict (existing event with same name+date is NOT historical — likely a duplicate of a real evaluated event; abort that group rather than silently overwriting real evaluation data).
- After commit, call the same `recomputeCycleResults` used elsewhere so cycle-level aggregates/snapshots stay in sync — don't hand-roll a separate aggregation.
- Bulk-created employees during import must go through the same insert path (defaults for department/function/sourceType) as any other manual employee, so they behave normally afterward (visible in employee lists, eligible for future cycles, etc.).
- Same "don't block on missing reference data" principle applies to the event's cycle: if no cycle's date range covers the event date, fall back to the current cycle (`getCurrentCycle`) with an info message rather than a hard error — only error out if there is truly no cycle at all in the system (nothing to fall back to).
- The pre-import preview dialog must show enough detail for a non-technical admin to trust the commit blindly: per-event action (create/update/conflict) with a legend, resolved cycle name + whether it was a fallback, new-employee names that will be auto-created, and summary counts (rows/matched/new employees/events/participants) — not just pass/fail.

**Debugging trap:** if you seed-reset the dev DB mid-session, any JWT captured *before* the reset embeds a now-nonexistent user id. Replaying it against a POST that eventually calls `recomputeCycleResults(cycleId, userId)` fails with a `quarterly_results_closed_by_user_id_users_id_fk` violation — looks like a real bug in the recompute/import code but is actually just a stale token. Always re-login for a fresh token after any DB reset before re-testing.

**Duplicate detection is name-based, so it under-matches:** the original "conflict" check only caught duplicates when the incoming event's name+date exactly matched an existing event. Real duplicates commonly have divergent names for the same race (e.g. "42K – Florianópolis – 2026" manually created vs. "42K DE FLORIPA" imported) and were silently created as separate events. Fix: for every `create`-action group, also compute date-overlap candidates (existing events whose date range overlaps, regardless of name) and surface them to the admin as `overlapCandidates` with a `groupKey`; let the admin optionally supply `linkOverrides: {groupKey: existingEventId}` in the commit payload to link into an existing event instead of creating a new one. Overlap ≠ duplicate — many same-day events are genuinely different races (different cities); this is a *review prompt*, not an auto-merge.

**Calibration entered on a historical event is a silent no-op:** entering per-criterion calibrations (`POST /calibrations`) and closing/confirming a historical event (`isHistorical=true`) stores the calibration rows fine, but the final score never changes — `recomputeCycleResults`/`computeEventTeamResult` skip straight to `importedScore` for historical events (see above), so the calibration data is orphaned. `PATCH /events/:id` only touches name/dates/location/status, never `isHistorical`.

**Direct score/notes correction (no CSV re-import needed):** `PATCH /events/:id/historical-result` (admin/rh only) directly edits `importedScore` (0-100) and/or `importedNotes` on an `isHistorical=true` event, then calls `recomputeCycleResults` — same recompute call the CSV import path uses, no other cache/snapshot to refresh. Rejects non-historical events (400) and out-of-range/non-numeric scores. There's also a matching inline edit panel on the event-detail page for admin/rh (read-only notes display for other roles). Recompute only affects cycle aggregates if the event also has `resultsConfirmed=true` — a score edit on an unconfirmed historical event updates the event's own display but won't move ranking/dashboard snapshots.

**Merging an already-created duplicate pair:** for duplicates that already exist as two separate event rows (not caught at import time), added a generic `POST /events/:id/merge` (admin-only, body `{mergeEventId}`) that keeps `:id` and absorbs+deletes `mergeEventId`: backfills only the *empty* fields on the kept event (clientName/location/city/state/externalId) from the merged one, migrates `event_participants` (onConflictDoNothing) and re-points `absences.eventId` (no cascade on that FK, so it must be reassigned before delete or the delete fails), then deletes the merge event and recomputes both events' cycles. Hard-blocks if the event being deleted already has evaluations/calibrations/conformity/employee_event_results rows, to never silently discard real scored data — that case needs manual resolution instead.

**Historical events skip evaluator assignment entirely:** the "Critérios e Avaliadores (RH)" panel (weight/evaluator-per-area config) only makes sense for events scored live through the app; it's hidden for `isHistorical` events (score already came from import, no evaluators needed). In its place, the event-detail page shows the historical note editor + the general event comments thread — this is the one place per event where those two live for historical events (not duplicated elsewhere on the page).
